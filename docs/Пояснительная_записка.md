## 1. Описание предметной области

«Бантуми» (Kalah) — стратегическая настольная игра из семейства Манкала. Рассчитана на двух игроков. Доска состоит из 12 малых лунок (по 6 на каждой стороне) и двух Калахов — хранилищ игроков.

### Правила игры

В начале в каждой лунке N камней (настраивается, 3–6). Игроки ходят поочерёдно: выбирают лунку на своей стороне и раскладывают камни по одному против часовой стрелки:

- **Бонусный ход** — последний камень попал в свой Калах: игрок ходит снова
- **Захват** — последний камень попал в пустую лунку своей стороны: забираются все камни из противоположной лунки противника
- **Конец игры** — у одного из игроков кончились камни: каждый забирает оставшиеся свои в Калах
- **Победитель** — у кого в Калахе больше камней

## 2. Архитектура приложения

Приложение построено по принципу **клиент-сервер**:

```
┌──────────────────────────┐         HTTP/JSON
│   frontend/main.py       │  ─────────────────▶  ┌──────────────────────────┐
│   Python + Tkinter       │                       │  backend/ (Spring Boot)  │
│                          │  ◀─────────────────  │                          │
│  BantumiApp              │    GameState / JSON   │  GameController          │
│  ├── _draw_board()       │                       │  GameService             │
│  ├── _on_pit_click()     │                       │  GameState / MoveResult  │
│  └── requests.post/get() │                       │                          │
└──────────────────────────┘                       └──────────────────────────┘
```

### 2.1 Бэкенд — Java / Spring Boot (Хусейнов А.)

**Слои приложения:**

| Слой | Класс | Описание |
|------|-------|---------|
| Controller | `GameController` | Принимает HTTP-запросы, возвращает JSON |
| Service | `GameService` | Вся игровая логика, хранит состояние |
| Model | `GameState` | DTO: доска, игрок, флаги, счётчик |
| Model | `MoveResult` | DTO: результат хода (valid, bonusTurn, captured) |

**REST API:**

| Метод | URL | Описание |
|-------|-----|---------|
| GET  | `/api/v1/game/state`        | Текущее состояние |
| POST | `/api/v1/game/new?stones=4` | Новая партия |
| POST | `/api/v1/game/move/{pit}`   | Ход из лунки |
| POST | `/api/v1/game/undo`         | Отмена хода |

**Структура доски (`board[14]`):**
```
Индексы 0–5   → лунки Игрока 1
Индекс  6     → Калах Игрока 1
Индексы 7–12  → лунки Игрока 2
Индекс  13    → Калах Игрока 2
```

**Ключевые алгоритмы в `GameService`:**

*Раскладывание камней:*
```
stones = board[pit]; board[pit] = 0
current = pit
while stones > 0:
    current = (current + 1) % 14
    if current == opponentKalah: continue
    board[current]++; stones--
```

*Проверка захвата:*
```
if ownsPit(current) and board[current] == 1 and board[12 - current] > 0:
    board[myKalah] += board[12 - current] + 1
    board[12 - current] = 0; board[current] = 0
```

*Undo:* перед каждым ходом в `ArrayDeque` сохраняется снимок состояния (массив из 18 int). При вызове `undo()` снимок восстанавливается.

### 2.2 Фронтенд — Python / Tkinter (Старостин М.)

**Класс `BantumiApp`** — единственный класс приложения.

| Метод | Описание |
|-------|---------|
| `_draw_board()` | Полная перерисовка Canvas после каждого ответа от API |
| `_draw_pit()` | Рисует лунку: овал, число, декоративные точки, hover |
| `_draw_kalah()` | Рисует Калах с индикатором активного хода |
| `_on_pit_click(pit)` | POST `/move/{pit}` → обновление UI |
| `_new_game()` | POST `/new` → сброс доски и журнала |
| `_undo()` | POST `/undo` → откат хода |
| `_fetch_state()` | GET `/state` → начальная загрузка при старте |

**Взаимодействие с API:** библиотека `requests`, таймаут 3 сек. Сетевые ошибки (`RequestException`) перехватываются отдельно от ошибок UI.

**Кнопки** реализованы через `Frame + Label` вместо `tk.Button` — единственный надёжный способ задать цвет на macOS.

## 7. Заключение

Приложение реализует полную логику игры «Бантуми» в архитектуре клиент-сервер. Java-бэкенд обеспечивает строгую валидацию ходов и хранение состояния через REST API. Python-фронтенд предоставляет интуитивный графический интерфейс. Разделение на два независимых компонента даёт возможность в будущем подключить другой клиент (мобильное приложение, веб-интерфейс) без изменений в логике.
